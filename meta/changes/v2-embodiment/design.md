# Design: v2-embodiment

## Approach

Everything in v2 flows through interfaces that already exist in `src/contracts.ts`; no contract changes are expected. The work is asset production, integration, and tuning, so the shape is: one sequential asset track (the bust is a hard dependency for everything else), then a short parallel wave.

## Asset track (sequential, first)

1. Source (decided by res.head-asset-alternatives, superseding res.asset-sourcing's MakeHuman-first ranking): primary is ICT-FaceKit Light (MIT, verified), the only surveyed candidate bundling a real quad human head with documented UVs and 53 ARKit expression blendshapes as in-repo data, with licence terms that survive sublicensing. Backup: MPFB2 (CC0 verified; base mesh plus 34 bundled FACS expression units). Correction recorded: the earlier "MPFB2 ships faceunits01 with 54 ARKit units plus visemes" claim was wrong; those are name lists in faceservice.py resolved from a separately installed target pack (52 shapes, licence unverified at download time), so the backup route uses only the bundled CC0 data. Hard requirements either way: under the accepted GLB delivery target of 1.5 MB post-Meshopt (dec.performance-budget), UVs usable by the projective text-skin mapping (face re-unwrapped to a dedicated island).
2. Author the 27 rig-schema morphs (15 visemes + 12 expressions; res.morph-authoring corrected the earlier count of 24). Neither route ships native visemes: author the 15 RIG_VISEME_MORPHS by compositing ARKit/FACS expression deltas (weights in specs/morph-authoring-detail.md) in a committed `@gltf-transform/core` script under `tools/asset-pipeline/`; sculpt any donor-less morphs in Blender as the fallback only. The pipeline keeps two artifacts: a full-fidelity intermediate that RETAINS every source delta (all 53 ICT ARKit shapes plus any authored shapes) and the composition recipe, committed or reproducibly regenerable; and the shipped GLB, pruned to the 27 canonical targets for budget. Retaining the source deltas makes a later expressiveness bump a re-wire, not a re-author. `validateRig` is the acceptance oracle and must assert all 27 canonical targets are present and reachable in the shipped GLB.
3. Run `tools/asset-pipeline/optimize.ts` (Meshopt + KTX2), then re-run `validateRig` (compression can strip or damage morph targets); record the exact command in the pipeline README.
4. Asset delivery decision: commit the optimized GLB (simplest, repo grows ~0.5 MB) versus fetch-on-build. Default: commit it; record dec.default-asset-delivery.

## Parallel wave (after asset lands)

- Demo integration: engine option `avatarUrl` default points at the bust; placeholder stays as documented fallback when load fails or URL is empty.
- Viseme e2e (decided by res.viseme-provider-format + res.local-tts-dev): two fixtures. (1) A strictly Polly-shaped viseme speech-mark JSONL fixture, GENERATED by a committed dev-only espeak-ng script (millisecond phoneme events mapped espeak -> Polly's 17-symbol en-US alphabet; no invented symbols, so the parser only ever accepts data a real provider could emit), replayed through SpeechEngine -> MotionEngine with a mock clock, asserting blendshape weights at keyframes. Polly's alphabet cannot reach `viseme_nn`, so this path asserts the 14 reachable morphs. (2) A canonical VisemeFrame timeline fixture injected at the adapter seam, asserting all 15 canonical morphs are drivable in MotionEngine, closing the `viseme_nn` gap found in res.local-tts-dev. espeak-ng is GPLv3 but remains a dev-time tool outside the shipped package; its generated timeline is treated as data (reasoning and caveats in res.local-tts-dev). Azure/Google/ElevenLabs rejected for the fixture: over-specification or no viseme output.
- Text-skin fit: tune `DEFAULT_GRID` density and emissive ramp against the bust UVs; screenshot-based manual check in the demo, numeric check on non-black pixel fraction like the v1 probe.

## Expressiveness expansion path

Default is 27 morphs for the embodiment milestone; `RIG_EXPRESSION_MORPHS` (12) and the `Expression` union are the frozen contract. Because the pipeline retains every source delta in the intermediate (asset track step 2), the authoring cost of new shapes is low. Two distinct deferred paths, do not conflate them: (1) Expand the canonical vocabulary via a scoped `dec.expression-vocab` supersede that edits the `Expression` union, `RIG_EXPRESSION_MORPHS`, and `EXPRESSION_MAP` (`src/motion/expressions.ts`) together in the SAME change, gated on measured size headroom and something actually driving the new shapes. (2) Expose user-authored custom emotes as an additive optional extension surface (API to be designed; NOT edits to the frozen 12-shape contract), deferred like the other future seams; the retained deltas and recipe are what make it cheap to build later. Either path wires its motion consumer in the same change, so no orphan morphs (shapes that exist but nothing drives).

## Future seams (recorded, not implemented in v2)

From the seam audit (res.seam-audit): the owner's future directions are reachable without breaking the shipped surface, but only via extension interfaces or optional members; adding required methods to `TTSAdapter`/`AudioEngine` would break implementors. Deferred seams, in likely order: (1) `AudioEngine` PCM/buffer connection path for a local in-browser TTS engine (Kokoro/kokoro-js, Apache-2.0, is the future mode-2 candidate per res.local-tts-dev); (2) generic animation trigger on `MotionEngine` playing `LoadedAvatar.animations` clips (data already loaded, currently unused); (3) shader-override surface on `VFXEngine` (emissive intensity, glyph colour, metalness/roughness; currently hardcoded in `src/shaders/materials.ts`); (4) engine-level `TextSkinEngine` injection for alternative scroll modes and reactive text. None of these gate v2.
