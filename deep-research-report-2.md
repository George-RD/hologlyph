## The recommended rendering stack

If the goal is a library that websites can trust, the best current default is a **Three.js core** with a **WebGPU-first, WebGL-safe** renderer posture. Three.js gives you the specific primitives this concept needs out of the box: glTF loading, clip-based animation playback, dynamic textures, custom materials, and a renderer path that can target WebGPU when available and fall back to WebGL 2 when it is not. That combination is unusually well-matched to a product that needs both high-end material effects and broad embed-ability. ţłÇciteţłéturn31view0ţłéturn20search1ţłéturn27view5ţłéturn28view0ţłü

For the ÔÇťskin made of ASCII and codeÔÇŁ concept, the smart default is to treat the surface text as a **dynamic texture source**, not as tens of thousands of individual glyph meshes. Three.js `CanvasTexture` is built specifically to create a texture from a canvas element, and `OffscreenCanvas` lets canvas work run away from the DOM and inside a worker context. That means you can render lorem ipsum, API-provided text, code streams, or terminal-like rows into a canvas, animate the text flow there, then project that animated surface onto the bust through a normal UV layout. This is the most practical route to keeping the face topologically precise while still letting the text visibly scroll and update in real time. ## The asset and animation pipeline

The asset format should be **GLB/glTF**, full stop. Khronos positions glTF as a format for efficient transmission and loading of 3D scenes and models, and the format natively supports the pieces you need: meshes, textures, skins, morph targets, and animations. For a library product, that matters because you are not just building one head ÔÇö you are building a pipeline for multiple default heads, future skins, motion packs, and possibly third-party avatars later. ţłÇciteţłéturn27view8ţłéturn20search9ţłü

For the two initial defaults ÔÇö your male and female busts ÔÇö the most important production decision is that they should share the **same rig and naming schema** even if they are visually distinct. In practice, that means identical skeleton semantics where applicable, identical morph target names for mouth and expression shapes, identical material slot conventions, and a common animation vocabulary. A shared schema is what lets one lip-sync system, one gesture library, and one behavior engine drive both defaults cleanly. This is not a glTF requirement by itself, but it is the natural design consequence of glTFÔÇÖs support for reusable animation/morph infrastructure and VRMÔÇÖs push to standardize pose, facial expression, and gaze operations. ţłÇciteţłéturn27view8ţłéturn26view0ţłéturn26view1ţłü

The most useful facial rig target for your use case is a **blendshape-heavy rig with procedural overrides**, not a bone-only face. VRM is especially relevant here as a design reference even if you do not ship VRM as the initial format: it standardizes facial expressions, lip-sync presets such as `a-i-u-e-o`, blink behavior, and gaze direction, and it explicitly treats pose, facial expression, and gaze as first-class operations. That makes VRM a strong conceptual template for your internal schema. My recommendation is to keep your default busts as custom GLBs to limit complexity in v1, but make the internal expression/gaze vocabulary deliberately ÔÇťVRM-likeÔÇŁ so that bring-your-own-avatar support can expand later with less refactoring. Three-vrm also documents WebGPU compatibility on current Three.js, which is useful if you later support imported VRM assets. ţłÇciteţłéturn26view0ţłéturn26view1ţłéturn26view2ţłü

For payload size and startup speed, you should bake optimization into the pipeline from day one. KhronosÔÇÖs `KHR_texture_basisu` extension adds KTX v2 with Basis Universal supercompression for more efficient texture transmission and lower GPU memory footprint, and Three.js provides `KTX2Loader` for transcoding those textures at runtime. On the geometry/animation side, `EXT_meshopt_compression` works at the bufferView level and supports not just geometry but also morph targets and animation data. `gltfpack` is particularly attractive because it does more than compression: it quantizes geometry, resamples animations, merges meshes to reduce draw calls, prunes redundant nodes, and can emit Meshopt-compressed output. glTF TransformÔÇÖs docs also note that Meshopt decoding is considerably faster than Draco decoding, while cautioning that compression alone does not improve framerate; to improve frame rate you still need fewer vertices and fewer draw calls. ţłÇciteţłéturn27view1ţłéturn27view4ţłéturn27view2ţłéturn27view6ţłéturn27view7ţłü

That last point is crucial for this project. A talking head feels ÔÇťfancy,ÔÇŁ but the actual 3D scene is small, which means your main performance enemies are usually **oversized textures, heavy post-processing, and unnecessary per-frame CPU work**, not the raw existence of a head model. ## Speech, lip sync, gaze, emotion, and gesture behavior

For lip sync, the best architecture is to separate **speech generation** from **facial driving data**. The most powerful option in the sources reviewed is Azure Speech, because AzureÔÇÖs speech synthesis viseme flow can emit viseme IDs, SVG animation, or 3D blend-shape data; MicrosoftÔÇÖs documentation states that its 3D blend-shape JSON is frame-based at 60 FPS and that each frame contains an array of 55 facial positions. That is unusually close to what a real-time web avatar library wants, because it can drive a facial rig directly instead of forcing you to infer mouth motion from waveform amplitude. ţłÇciteţłéturn23view0ţłéturn24view0ţłéturn24view1ţłéturn24view2ţłü

Amazon Polly is the next-strongest option for a portable production path. PollyÔÇÖs speech marks can return `sentence`, `word`, `viseme`, and `ssml` metadata, and its output format is line-delimited JSON with millisecond timestamps from the beginning of the audio stream. That is enough to create a solid viseme-to-blendshape adapter in the browser. Google Cloud TTS is still useful, but in the official docs reviewed here the most directly relevant synchronization primitive is **SSML timepoints via `<mark>`**, which is excellent for coarse synchronization and caption timing but is not the same thing as a native viseme event stream. In other words: if precise mouth motion is a marquee feature, Azure is the cleanest fit, Polly is strong, and Google Cloud is better suited to coarser temporal alignment unless you add your own phoneme/viseme layer. ţłÇciteţłéturn23view1ţłéturn23view2ţłéturn23view3ţłü

Your realism instinct about eye contact is exactly right. Social-interaction research consistently treats gaze as a conversation-regulation signal rather than as a fixed ÔÇťstare at the userÔÇŁ target. Ho, Foulsham, and Kingstone found that eye gaze can help signal both the end and the beginning of speaking turns. The same applies to nodding. Research on head movement shows that nods are not one generic gesture. ## Scroll choreography and page integration

The scroll behavior should be implemented as a **behavioral timeline**, not as a CSS gimmick. CSS scroll-driven animations are real and useful, but the web-features explorer still marks them as ÔÇťLimited availability,ÔÇŁ with Baseline blocked by Firefox as of mid-2026. MDNÔÇÖs scroll-driven animation docs are valuable, but in this project they should be treated as an optional enhancement, not as the primary control system. The cross-browser core should instead derive a normalized scroll progress in JavaScript and drive your animation state machine and mixer with that value. ţłÇciteţłéturn37view0ţłéturn21view4ţłéturn27view5ţłü

That approach also aligns better with the kinds of effects you want. `IntersectionObserver` gives you an efficient, browser-optimized way to know when the host section is entering or leaving the viewport, and `requestAnimationFrame()` gives you a display-synced update loop whose timestamp should be used for time-correct motion on high-refresh-rate screens. Put those together and you can trigger ÔÇťemerge from pool,ÔÇŁ ÔÇťtrack scroll progress,ÔÇŁ ÔÇťpause when mostly offscreen,ÔÇŁ and ÔÇťsink away on exitÔÇŁ without tying your logic to a not-yet-universal CSS feature. `ResizeObserver` should also be part of the host integration so the canvas can respond to container-level layout changes rather than assuming full-window behavior. ţłÇciteţłéturn22view5ţłéturn28view4ţłéturn22view4ţłü

A practical choreography model is: when the componentÔÇÖs section first becomes visible, it transitions from `hidden` to `emerging`; once settled, it goes to `idle-listening`; if the page or app pushes text for speech, it moves to `speaking`; if the text stream is paused or a long latency gap appears, it can briefly enter `thinking`; when the component leaves the viewport, it either freezes in place or returns to `submerging` before cleanup, depending on the siteÔÇÖs visual language. Because `Page Visibility API` tells you when the document itself becomes hidden, you can also suspend the render loop, audio analysis, and any idle behaviors when the tab is backgrounded. ţłÇciteţłéturn22view3ţłéturn28view4ţłü

The ÔÇťmove with scrollÔÇŁ concept should be authored sparingly. Big viewport-scale motion is visually striking, but MDNÔÇÖs `prefers-reduced-motion` guidance emphasizes that non-essential motion can trigger discomfort for users with vestibular sensitivities. In product terms, that means pool emergence, idle sway, side-to-side drift, and dramatic camera-follow behaviors should all have a reduced-motion variant that either shortens the transition, fades instead of travels, or disables the effect altogether. This is especially important for a talking head, because users may encounter it on pages they did not intentionally visit for rich animation. ţłÇciteţłéturn28view5ţłü

## Packaging, performance, and rollout

The library should ship as a **single custom element with adapters**, not as a React-only or Vue-only widget. A `<talking-head>` custom element gives you browser-native mounting, teardown, and configuration, and then you can offer thin wrappers for React, Vue, Svelte, and plain HTML. That keeps the core renderer and behavior engine independent from framework churn while still fitting modern site stacks. The element should lazy-load its renderer bundle with dynamic `import()`, initialize only when it is near the viewport, and expose a declarative attribute/property API for the most common cases. ţłÇciteţłéturn22view0ţłéturn22view1ţłéturn22view2ţłéturn22view5ţłü

A good public API would be semantic rather than low-level. In other words, website authors should say things like `mode="listen"`, `speak(text)`, `setEmotion("friendly")`, `setScrollProgress(x)`, `setTextSkinSource(source)`, and `setVoiceAdapter(adapter)`, rather than manipulating morph targets or animation actions directly. Under the hood, the library can still expose advanced hooks for custom TTS, custom viseme streams, host analytics, or custom face-text generators. That separation is what will make the product usable by design teams and front-end teams, not just graphics specialists. The feasibility of this packaging model is strengthened by the browser-native custom-element lifecycle and by the fact that the loading and rendering subsystems can be split into on-demand bundles. ţłÇciteţłéturn22view0ţłéturn22view1ţłéturn22view2ţłü
